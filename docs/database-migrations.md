# Database Migrations

The project already uses Alembic for schema changes.

Current migration components:

- `backend/alembic.ini` configures Alembic.
- `backend/alembic/env.py` reads `DATABASE_URL` from application settings and
  sets `target_metadata` from the SQLAlchemy model metadata.
- `backend/alembic/versions/` contains the migration history.
- FastAPI startup does not call `Base.metadata.create_all()` and does not
  migrate the database automatically.

Tests may create temporary in-memory schemas with SQLAlchemy helpers, but that
is test setup only and is not the production migration path.

## Normal Workflow

After changing SQLAlchemy models, create a migration inside the backend
environment:

```bash
docker compose exec backend alembic revision --autogenerate -m "describe change"
```

Review the generated file in `backend/alembic/versions/` before applying it.
Autogenerated migrations can miss data backfills, indexes, constraints, or
custom SQL details.

Apply migrations explicitly:

```bash
docker compose exec backend alembic upgrade head
```

Check the current revision:

```bash
docker compose exec backend alembic current
```

## Mac With Remote Database

When the Mac uses an SSH tunnel to the VPS PostgreSQL database, migration
commands run from the Mac will affect the remote database. Only run them after:

- confirming `DATABASE_URL` points to the intended target;
- reviewing the migration file;
- committing or otherwise preserving the migration source;
- taking any backup appropriate for the risk of the schema change.

## Rules

- Do not edit files under `data/postgres` directly.
- Do not use manual database file copies as a migration mechanism.
- Do not rely on application startup to create or repair schema.
- Keep migration files free of real secrets, raw IBKR XML, and local paths that
  only exist on one machine.
